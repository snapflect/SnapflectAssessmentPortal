# SNAPFLECT ASSESSMENT PORTAL
## 023_SPRINT_01_IMPLEMENTATION_REVIEW_FRAMEWORK
**Version:** 1.0  
**Status:** APPROVED BASELINE  
**Purpose:** Quality Gate & Review Checklists for Cursor AI Generated Code  

### 1. PURPOSE
This document defines the strict review criteria that must be applied to all code generated by Cursor AI before it is approved for merging. It serves as the definitive Quality Gate for Sprint 01 and beyond. No code may bypass these checklists.

---

### 2. ARCHITECTURE REVIEW CHECKLIST

**Layer Separation & Responsibilities**
- [ ] **Controllers:** Do not contain business logic. They only handle HTTP requests, validate via Form Requests, call Services, and return Responses/Resources. Maximum size <= 300 lines.
- [ ] **Services:** Contain all business rules, orchestration, and workflow logic. No direct database queries (must use Repositories). No HTTP/Request logic. Maximum size <= 1000 lines.
- [ ] **Repositories:** Contain all database access logic. No business rules. Implement Interfaces.
- [ ] **Models:** Only represent database structure and Eloquent relationships. No business or workflow logic.
- [ ] **DTOs:** Data Transfer Objects are used between Controllers and Services. No raw arrays passed between layers.

**Domain & Data Integrity**
- [ ] **Soft Deletes:** `is_deleted`, `deleted_by`, `deleted_date` implemented and enforced. No physical deletes.
- [ ] **Auditability:** `created_by`, `created_date`, `modified_by`, `modified_date` populated correctly on mutations.
- [ ] **Tenant Isolation:** Organization-scoped queries strictly enforced where applicable (no cross-tenant data bleed).
- [ ] **Primary Keys:** UUIDs used for external references. Internal BIGINT IDs never exposed in API responses.

---

### 3. SECURITY REVIEW CHECKLIST

**Authentication & Authorization**
- [ ] **Sanctum Implementation:** Authentication leverages Laravel Sanctum exclusively. No leftover JWT logic.
- [ ] **RBAC Enforcement:** Policies and/or Middleware correctly enforce permissions mapping to `016_RBAC_MATRIX`.
- [ ] **Separation of Duties:** Roles cannot bypass their defined limits (e.g., Question Authors cannot approve own questions).

**Data Protection**
- [ ] **Passwords:** Stored securely using bcrypt hashing. Never logged, never returned in API payloads.
- [ ] **Input Validation:** Every write endpoint uses Laravel Form Requests. Strong typing and boundaries enforced.
- [ ] **Output Encoding:** API Resources filter sensitive fields (hidden attributes honored). Blade templates use `{{ }}` to prevent XSS.
- [ ] **Vulnerability Mitigation:** CSRF protection enabled (for web routes). SQL injection prevented (no raw queries without bindings).
- [ ] **Rate Limiting:** Global rate limiting (60/min default, 120/min authenticated) and endpoint-specific limits (e.g., Login 5/15m) applied.

---

### 4. PERFORMANCE REVIEW CHECKLIST

**Database & Queries**
- [ ] **N+1 Problem:** Eager loading (`with()`) used appropriately to prevent N+1 queries.
- [ ] **Indexes:** Foreign keys and frequently queried columns (e.g., `email`, `status`, `organization_id`) are indexed in migrations.
- [ ] **Pagination:** All list endpoints use pagination. No unbounded `get()` queries on potentially large tables.
- [ ] **Query Complexity:** Avoided massive joins. Complex aggregates pushed to derived services or background jobs where appropriate.

**Application Level**
- [ ] **Background Processing:** Long-running tasks (Emails, Audit aggregation) pushed to the Database Queue.
- [ ] **Caching:** Expensive and infrequently changing data (e.g., Roles/Permissions lists) utilizes Laravel Cache.

---

### 5. LARAVEL CODING STANDARDS CHECKLIST

**Conventions & Standards**
- [ ] **Naming:** `snake_case` for DB columns/tables. `camelCase` for variables/methods. `PascalCase` for Classes/Models.
- [ ] **PSR Compliance:** Code adheres to PSR-12 styling conventions.
- [ ] **Type Hinting:** Strict type hinting used for all method arguments and return types (PHP 8.4 features leveraged).

**Implementation Details**
- [ ] **Exception Handling:** Global exception handler captures and formats errors consistently. No raw stack traces exposed.
- [ ] **Configuration:** Environment variables (`.env`) used for all external connections and secrets. No hardcoded credentials.
- [ ] **Blade Views:** Follow component-based architecture where possible. Leverage Bootstrap 5 cleanly without messy inline CSS.

---

### 6. OPENAPI COMPLIANCE CHECKLIST

**Contract Fidelity**
- [ ] **Endpoint Match:** Route paths and HTTP verbs exactly match the OpenAPI specifications (`017A` - `017K`).
- [ ] **Payload Match:** Request Formats (JSON) exactly match the specified Request schemas.
- [ ] **Response Match:** Success and Error responses strictly follow the standardized envelope structure (`success`, `message`, `data`/`errors`).
- [ ] **Status Codes:** Correct HTTP status codes returned (200, 201, 400, 401, 403, 404, 422, 500).

---

### 7. DEFINITION OF DONE CHECKLIST

**Before Merge Approval, the following must be true:**
- [ ] **Code Complete:** All requested features for the story are implemented.
- [ ] **Tests Passing:** Unit, Feature, and Integration tests pass. Target coverage of 80%+ achieved.
- [ ] **Swagger Updated:** API documentation generated/updated to reflect the code.
- [ ] **Review Passed:** Architecture, Security, Performance, and Standards checklists (above) are fully validated.
- [ ] **Zero Critical Defects:** No blocking bugs or security vulnerabilities present.
- [ ] **Ready for Deployment:** Code successfully builds and migrates from scratch (`migrate:fresh --seed` works without errors).
