EXECUTIVE SUMMARY The Assessment Delivery Engine transforms a published assessment into a secure, auditable, immutable candidate assessment experience. This architecture governs: Assessment Launch Assessment Sessions Assessment Attempts Question Rendering Answer Capture Auto Save Resume Submission Attempt Locking Audit Trail Sprint 03 introduces the first real-time candidate-facing execution workflow within Snapflect. 2. ARCHITECTURAL PRINCIPLES Principle 1 – Snapshot-Based Delivery Candidates never consume a live assessment. Candidates always consume: Assessment Snapshot Source: assessment_snapshots Benefits: Historical integrity Version safety Legal defensibility Audit compliance Principle 2 – Immutable Submission After submission: No answer updates No timer changes No resume No attempt reopening Submitted attempts become immutable records. Principle 3 – Server-Authoritative Time The server owns time. Never trust: Browser timer Client clock JavaScript time calculations All timing decisions originate from backend services. Principle 4 – Full Auditability Every candidate action must generate an audit event. Examples: Session start Question viewed Answer saved Resume Submit Timeout Principle 5 – Single Active Attempt One candidate may have only one active attempt for a specific assessment version. 3. DOMAIN MODEL Assessment Snapshot         │         ▼ Assessment Session         │         ▼ Assessment Attempt         │  ┌──────┼────────┐  ▼      ▼        ▼ Questions Answers Events         │         ▼ Submission 4. CORE ENTITIES Assessment Session Represents candidate access authorization. Responsibilities: Launch validation Resume validation Expiration control Session security Assessment Attempt Represents the actual assessment execution. Responsibilities: Progress tracking Status management Timer tracking Submission control Attempt Question Represents a snapshot-derived question delivered to a candidate. Contains: Snapshot question metadata Display order Section assignment Candidate Answer Represents candidate responses. Supported answer types: Single Choice Multiple Choice True/False Short Text Long Text Numeric Attempt Event Represents immutable audit records. Examples: SESSION_STARTED ANSWER_SAVED QUESTION_VIEWED SESSION_RESUMED SUBMITTED Attempt Submission Represents final locked assessment submission. 5. SESSION LIFECYCLE Session Creation Trigger: Candidate clicks Start Assessment Workflow: Validate Assignment         ↓ Validate Published Snapshot         ↓ Validate Availability Window         ↓ Create Session Session States CREATED ACTIVE EXPIRED TERMINATED Session State Machine CREATED     ↓ ACTIVE  ┌──┴──┐  ↓     ↓ EXPIRED TERMINATED 6. ATTEMPT LIFECYCLE Attempt States NOT_STARTED IN_PROGRESS SUBMITTED EXPIRED ABANDONED LOCKED State Machine NOT_STARTED       ↓ IN_PROGRESS    ↓      ↓    ▼      ▼ SUBMITTED EXPIRED    ↓ LOCKED Rules: Submission immediately locks the attempt. Locked attempts are read-only. 7. QUESTION RENDERING ENGINE Questions must always originate from: Assessment Snapshot Never: Live Assessment Live Blueprint Live Question Bank Rendering Flow: Snapshot    ↓ Attempt Questions    ↓ Delivery API    ↓ Candidate UI 8. ANSWER LIFECYCLE Create Answer Triggered when: Candidate saves response Update Answer Allowed only while: Attempt Status = IN_PROGRESS Lock Answer Occurs during submission. State Machine: EMPTY   ↓ DRAFT   ↓ UPDATED   ↓ LOCKED 9. AUTO SAVE ENGINE Rule: Save every answer change Workflow: Candidate Changes Answer          ↓ API Save Request          ↓ Persist Answer          ↓ Create Audit Event          ↓ Return Success Requirements: Survive browser refresh Survive temporary network interruptions Recover on resume 10. RESUME ENGINE Resume must restore: Answers Current question Section progress Remaining time Navigation state Workflow: Candidate Login       ↓ Load Session       ↓ Load Attempt       ↓ Load Answers       ↓ Load Remaining Time       ↓ Continue Assessment 11. TIMER ENGINE Server Authority Timer controlled exclusively by backend. Stored values: started_at expires_at remaining_seconds Expiration Workflow Current Time &gt; Expires At              ↓ Auto Submit or Auto Expire Behavior determined by assessment configuration. 12. SUBMISSION ENGINE Workflow: Validate Session        ↓ Validate Attempt        ↓ Lock Answers        ↓ Create Submission        ↓ Update Attempt Status        ↓ Create Audit Event Result: Attempt = SUBMITTED Answers = LOCKED 13. ATTEMPT LOCKING After submission: No editing No answer changes No resume No reopening Allowed: Read-only access 14. SNAPSHOT CONSUMPTION Assessment Delivery always consumes: Published Snapshot Never: Live Assessment Structures Benefits: Version consistency Historical accuracy Legal defensibility Audit integrity 15. AUDIT ARCHITECTURE Every significant action generates an Attempt Event. Event Types SESSION_STARTED QUESTION_VIEWED ANSWER_SAVED ANSWER_UPDATED SECTION_CHANGED SESSION_RESUMED SUBMITTED EXPIRED AUTO_SUBMITTED Retention: Permanent No deletion permitted. 16. RBAC MATRIX Action Candidate Department Manager Organization Admin Platform Admin Launch Assessment Yes No No Yes Save Answer Yes No No Yes Resume Attempt Yes No No Yes Submit Attempt Yes No No Yes View Own Attempt Yes No No Yes View Tenant Attempts No Yes Yes Yes Force Expire Attempt No No Yes Yes 17. TENANT ISOLATION Every delivery entity must contain: organization_id Candidate access limited to: Own session Own attempt Own answers Organization Admin access limited to: Tenant attempts Tenant sessions Tenant submissions Only within the same organization. 18. CONCURRENCY CONTROL Prevent: Multiple Active Attempts Rule: One Candidate + One Assessment Version = One Active Attempt Enforced via service layer and database constraints. 19. FAILURE RECOVERY Supported scenarios: Browser refresh Browser crash Network interruption Session resume Auto-save recovery No candidate answers should be lost. 20. FUTURE EXTENSIBILITY Sprint 03 prepares for: Sprint 04 – Evaluation Engine Sprint 05 – Analytics Engine Sprint 06 – Certification Engine Sprint 07 – Runtime AI Proctoring 21. SUCCESS CRITERIA Sprint 03 is successful when: Assessment launches successfully Session created successfully Attempt created successfully Questions render correctly Answers save correctly Auto-save works Resume works Timer works Submission works Audit events generated Attempt locks correctly Without: Cross-tenant access Timer manipulation Assessment mutation Answer loss Unauthorized access 22.
