# 213 Results Dashboard Rulebook

**Document Version:** 1.0
**Phase:** Sprint 08 Planning
**Domain:** Results & Reporting Engine

---

## 1. OBJECTIVE

The objective of the Results & Reporting domain is to consume the data generated by the Sprint 07 Scoring Engine and present it effectively to multiple stakeholders. This rulebook establishes the business logic, visibility constraints, and governance models for dashboards, analytics, leaderboards, and certification.

---

## 2. CANDIDATE VISIBILITY RULES

Candidates must only see what the Assessment Blueprint permits them to see.

### 2.1 Result Visibility Rules
- **Rule 2.1.1:** Candidates can only view results that are in a `PUBLISHED` state. Results in `PENDING`, `CALCULATING`, or `READY` states are invisible.
- **Rule 2.1.2:** A `score_visibility` flag on the assessment blueprint determines if the raw score and percentage are shown to the candidate.
- **Rule 2.1.3:** A `pass_fail_visibility` flag determines if the final `PASS` or `FAIL` verdict is displayed.
- **Rule 2.1.4:** If the blueprint is configured to hide all scores, the candidate only receives a generic completion acknowledgement.

### 2.2 Score Breakdown Rules
- **Rule 2.2.1:** If `show_question_breakdown = true`, the candidate may view which specific questions they answered correctly or incorrectly.
- **Rule 2.2.2:** If `show_correct_answers = true`, the candidate is allowed to see the actual correct answer key alongside their incorrect responses.

### 2.3 Competency Charts
- **Rule 2.3.1:** Competency sub-scores are visualized using radar/spider charts or bar graphs.
- **Rule 2.3.2:** Competency visibility is governed by a `show_competencies` flag. 

### 2.4 Historical Attempts
- **Rule 2.4.1:** Candidates have a dedicated portal to view their historical attempts across all published assessments.
- **Rule 2.4.2:** Historical views must respect the blueprint rules that existed at the exact time the attempt was taken (`assessment_snapshot_id`).

---

## 3. ADMIN ANALYTICS & VISIBILITY

Administrators require deep visibility and aggregate insights, completely bypassing candidate visibility constraints.

### 3.1 Organization Dashboards
- **Rule 3.1.1:** Admins can view all candidate attempts, including those currently in `CALCULATING` or `READY` states.
- **Rule 3.1.2:** Admins have full access to raw scores, percentages, and `pass_fail_status` regardless of candidate visibility settings.

### 3.2 Aggregate Analytics
- **Rule 3.2.1:** Admins can view aggregate pass/fail rates per Assessment.
- **Rule 3.2.2:** Admins can view average competency performance across cohorts to identify systemic knowledge gaps (e.g., "70% of candidates fail the CSS competency").
- **Rule 3.2.3:** Item Analysis: Admins can view pass rates per specific question to identify potentially flawed or confusing questions.

---

## 4. LEADERBOARDS ENGINE

Leaderboards introduce gamification and cohort ranking.

### 4.1 Leaderboard Governance
- **Rule 4.1.1:** A leaderboard is only generated if the assessment blueprint explicitly enables `leaderboard_enabled = true`.
- **Rule 4.1.2:** Candidates can opt-out of public leaderboard visibility via a privacy toggle in their user settings. Opted-out users appear as "Anonymous Candidate".
- **Rule 4.1.3:** Ranking is determined by: 1) `overall_score` (descending), followed by 2) `time_taken` (ascending) as a tie-breaker.

---

## 5. CERTIFICATION ENGINE

Certificates provide formal verification of successful completion.

### 5.1 Certificate Eligibility
- **Rule 5.1.1:** Certificates are exclusively generated for attempts with a `pass_fail_status = PASS`.
- **Rule 5.1.2:** The assessment blueprint must have `certification_enabled = true`.
- **Rule 5.1.3:** Certificates are issued automatically upon the result transitioning to the `PUBLISHED` state.

### 5.2 Certificate Immutability
- **Rule 5.2.1:** A generated certificate is permanently tied to a specific `result_version`.
- **Rule 5.2.2:** If a score is recalculated by an admin (creating `Version N+1`), and the new version falls below passing, the original certificate is marked `REVOKED`.
- **Rule 5.2.3:** Certificates feature a unique verification URL for third-party validation.

---

## 6. RESULT IMMUTABILITY & AUDIT

The presentation layer must respect the immutable nature of the Scoring Engine.

- **Rule 6.1.1:** The Dashboard never recalculates a score on the fly. It strictly reads the latest `assessment_results` row.
- **Rule 6.1.2:** When rendering historical graphs, the UI displays the `result_version` string to ensure transparency if an admin override occurred.
- **Rule 6.1.3:** The complete JSON `question_ledger` generated in Phase 7 is exposed in the Admin UI as an expandable "Audit Trail" view for every single attempt.
