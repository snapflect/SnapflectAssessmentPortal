# User Provisioning and Invitation Architecture

This document describes the unified user invitation and onboarding workflow within the Snapflect Assessment Portal. This architecture ensures that all users, whether provisioned dynamically during workspace creation or manually via the tenant's admin dashboard, traverse the exact same secure onboarding pipeline.

## 1. Core Concepts

The user onboarding pipeline is built on a "Pending-to-Active" state machine secured by Laravel's native Password Broker.

- **Pending User**: A skeleton `User` record created with a random, temporary password and the `PENDING` status. They are assigned a role but cannot authenticate.
- **Secure Token**: A unique, cryptographically secure string generated by the `Password::broker()` with a strict 72-hour expiration window.
- **Claim URL**: A dynamically generated URL mapped to the tenant's exact subdomain containing the token and user's email.
- **Active User**: After successfully submitting the Claim Account form (providing their name and a new password), the token is consumed, and the user status switches to `ACTIVE`.

## 2. The Universal Pipeline (`UserInvitationService`)

To prevent fragmentation between automated system events and manual UI actions, *all* invitations are routed through a single backend engine: `UserInvitationService`.

### The `inviteUser` Workflow:
1. **Role Validation**: Validates that the requested `role_code` exists and belongs to the specified organization. Throws `EntityNotFoundException` on failure.
2. **Duplicate/Conflict Check**: 
   - If the email belongs to a different organization, throws a `TenantValidationException` (Cross-tenant breach prevention).
   - If the email belongs to the current organization and is `ACTIVE`, throws a `TenantValidationException` (Password overwrite prevention).
3. **Record Creation**: If the user does not exist, a new `User` record is created in the `PENDING` state.
4. **Role Assignment**: Assigns the requested role using `UserService::assignRole`, attaching the caller's ID (or the workspace creator's ID for queued jobs) for accurate audit trails.
5. **Token Generation**: Generates a 72-hour `Password::broker()` token.
6. **Email Dispatch**: Dispatches the branded `UserInviteMail`, injecting the dynamically generated Tenant URL.

## 3. Automation vs. Manual Flows

### A. Automated Workspace Provisioning
When a new organization is purchased via the Enterprise Onboarding Wizard:
- The system dispatches `ProvisionTenantUsersJob` in the background.
- This job iterates over the primary `contact_email` (assigning `ORG_ADMIN`) and any `pending_invites` collected from the wizard.
- It injects the `created_by` ID of the workspace creator to ensure audit logs correctly attribute these initial invites.

### B. Manual Dashboard Invites
When a tenant administrator clicks "Add User" in their RBAC dashboard:
- The UI sends a `POST /api/v1/security/users/invite` payload.
- The `UserController@invite` method rigorously validates tenant isolation: Non-platform admins can only pass an `organization_id` that strictly matches their own.
- The payload is then handed directly to the `UserInvitationService`.

## 4. The Claim Account Flow

1. The user clicks the link in their `UserInviteMail`.
2. They are directed to the Angular `ClaimAccountComponent` on the tenant's subdomain.
3. The UI presents a form requiring `first_name`, `last_name`, `password`, and `password_confirmation`.
4. The payload is sent to `POST /api/v1/auth/claim-account`.
5. The `AuthController@claimAccount` backend verifies the `Password::broker()` token.
6. Upon success:
   - The token is deleted to prevent reuse.
   - The user's status is updated to `ACTIVE`.
   - The user's name and hashed password are saved.
   - The server instantly generates and returns a JWT access token, auto-logging the user directly into their new enterprise dashboard.
