PART 3
ASSESSMENT EXECUTION, ATTEMPTS, SNAPSHOTTING, SCORING & REVIEW
This section defines the complete runtime architecture responsible for assessment execution, attempt management, answer collection, scoring, evaluation, review, and result generation.
This is the most critical subsystem of the platform because it governs the integrity of all assessment data.

40. ASSESSMENT SNAPSHOT ARCHITECTURE
Purpose
Protect historical assessment integrity.

Problem Statement
Question definitions may change over time.
Example:
Question 1
Correct Answer = A
User completes assessment.
Administrator later changes answer.
Correct Answer = B
Without snapshotting:
Historical results become invalid.

Mandatory Requirement
At assessment start:
Create immutable snapshot of:
Technology
Assessment
Sections
Question Banks
Questions
Options
Correct Answers
Scoring Rules
Passing Rules
Assessment Rules

Snapshot Storage
Snapshot shall remain unchanged forever.
Even if:
Question Changes
Answer Changes
Question Bank Changes
Assessment Changes
Technology Changes

Snapshot Usage
Result Calculation
Review
Reporting
Audit
Historical Analysis
Re-Evaluation

65. DATA INTEGRITY REQUIREMENTS
No scoring against mutable master data.
Scoring must use snapshots.
Submitted attempts are immutable.
Reviews are auditable.
Results are reproducible.
Historical reports must remain accurate.
