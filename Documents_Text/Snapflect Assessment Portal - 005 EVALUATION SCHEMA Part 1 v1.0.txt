EVALUATION_RULES
Purpose
Represents reusable evaluation rule definitions.
Rules are versioned through:
evaluation_rule_versions
Rule content is never stored in the master table.

RULE DEFINITION STRATEGY
Rules are stored as structured JSON.
Example:
{"passPercentage":70,"negativeMark":0.25}

VALIDATION SCHEMA STRATEGY
Optional validation schema:
{"required":["passPercentage"]}
Used by the Rule Engine before publication.

EFFECTIVE DATING STRATEGY
Supports:
Scheduled Rule ActivationCompliance WindowsFuture Rule Rollouts
Rules become active when:
current_timestamp >= effective_from
and
current_timestamp < effective_to
when effective_to exists.

EXECUTION ORDER STRATEGY
Rules execute in ascending order.
Example:
1 = NegativeMarking2 = WeightedScoring3 = PassPercentage
Ensures deterministic evaluation.

METADATA STRATEGY
metadata_json
supports future:
AI Evaluation ParametersCertification MetadataRule CategoriesAdvanced Scoring Extensions
without schema changes.

GOVERNANCE RULES
Rule Code Immutability
rule_code
is immutable after creation.
Changes require creation of a new rule entity.

Published Version Rule
Published versions are read-only.
Changes require creation of:
New Draft Version

Historical Reproducibility Rule
Evaluations must always reference the exact:
evaluation_rule_version_id
used at evaluation time.
