1. PURPOSE
The Governance Foundation Schema provides shared platform infrastructure services consumed by all application modules.
This schema establishes:
File Metadata Management
Generic Attachment Management
Configuration Management
Audit Logging
Activity Tracking
All subsequent modules will depend on one or more entities from this schema.

2. TABLES IN SCOPE
File Infrastructure
file_uploadsattachments

Configuration Management
system_settings

Governance & Compliance
audit_logsactivity_logs

3. ARCHITECTURE PRINCIPLES
3.1 File Storage Principle
Approved Architecture:
Metadata In DatabaseFiles In Storage
Database Stores:
File NameStored File NameFile PathChecksumFile SizeMIME TypeStorage ProviderLifecycle Status
Database Does Not Store:
Binary ContentImagesDocumentsVideos

3.2 Attachment Principle
Approved Architecture:
Generic Polymorphic Attachment Architecture
Supported Entity Types:
TechnologyAssessmentSectionQuestion BankQuestionReview
Future modules can attach files without schema redesign.

3.3 Configuration Principle
Approved Architecture:
Scoped Configuration Architecture
Supported Scopes:
GlobalTechnologyAssessmentSectionQuestion Bank
Configuration Resolution Order:
Section↓Assessment↓Technology↓Global

3.4 Audit Principle
Approved Architecture:
Generic Polymorphic Audit Architecture
Allows auditing of any entity in the system.

3.5 Activity Principle
Activity logs are intentionally separate from audit logs.
Audit Log:
System Change
Activity Log:
User Action

6. FILE_UPLOADS DESIGN
Purpose
Stores uploaded file metadata.

Core Attributes
uuidfile_namestored_file_namefile_pathfile_extensionmime_typefile_sizechecksumstorage_providerstatus

Supported Storage Providers
HostingerLocalAzure BlobAWS S3

Supported Status Values
UploadedActiveArchivedQuarantinedDeleted

Business Rules
Physical files remain outside database storage.
Checksum validates file integrity.
Quarantined files cannot be attached.
Deleted files are soft deleted.

7. ATTACHMENTS DESIGN
Purpose
Associates uploaded files with business entities.

Architecture
Polymorphic Attachment Model

Core Attributes
file_upload_identity_typeentity_idattachment_typedisplay_orderversion_numberis_current_versionparent_attachment_id

Versioning Strategy
Supported.
Example:
QuestionImage_v1      ↓QuestionImage_v2      ↓QuestionImage_v3

Business Rules
Historical versions are retained.
Only one active version exists at a time.
Replacing an attachment creates a new version.

9. AUDIT_LOGS DESIGN
Purpose
Stores immutable system audit history.

Architecture
Generic Polymorphic Audit Architecture

Core Attributes
entity_typeentity_identity_versionaction_typeold_value_jsonnew_value_jsonperformed_bycorrelation_id

Supported Actions
CreateUpdateDeletePublishApproveRejectAssignSubmitReview

Correlation ID Purpose
Tracks related actions across:
API RequestsWorkflow ExecutionBulk ImportsBulk Exports

Business Rules
Audit logs are immutable.
Audit logs are never deleted.
Minimum retention period = 7 years.
Full assessment snapshots are prohibited.
Metadata changes only.
